The title “Software Developer” increasingly describes an activity, not the accountability. If AI agents can turn intent into code, tests, refactorings, documentation, and deployment steps, then the scarce human contribution is no longer the ability to type implementation details quickly. The scarce contribution is the ability to shape, constrain, verify, and evolve systems that create measurable value.

That is why I like the phrase **Systems Value Engineer**.

It preserves the seriousness of engineering. It keeps the focus on systems rather than isolated code. And it makes value creation explicit, which is exactly where the profession should move as coding becomes more automated.

## The old title points at the artifact

“Software Developer” made sense when software was the primary artifact and writing it was the bottleneck. The title pointed at what the person produced: software.

That framing is starting to feel too narrow.

A modern delivery system contains much more than source code:

- product intent
- architectural constraints
- domain rules
- security controls
- regulatory obligations
- test suites
- telemetry
- release workflows
- operating procedures
- support feedback
- user behaviour
- cost signals
- agent instructions
- institutional knowledge

Code is still important. But it is no longer enough to say that the job is to “develop software.” The job is to ensure that a messy sociotechnical system reliably produces useful outcomes.

The better title should point at that accountability.

## Why not just keep “Software Engineer”?

“Software Engineer” is still a strong title. It implies discipline, design, tradeoffs, correctness, and responsibility. It will not disappear quickly, especially in large enterprises, regulated industries, platform teams, and safety-conscious environments.

But it still centres the word “software.” That is both its strength and its limitation.

The next version of the role will spend more time on questions like:

- What should the system do?
- How do we know it did the right thing?
- Which constraints must never be violated?
- Which parts can be delegated to agents?
- What evidence proves that the change is safe?
- How will the system fail?
- How will humans recover it?
- Which business capability is improved?
- Which metric moved?

Those are engineering questions, but they are not only software questions.

## Why “Systems Value Creator” is close but not quite enough

“Systems Value Creator” captures the important shift. It says the role is not just about writing code. It says the goal is value through systems.

But as a formal title, “Creator” is too soft. It can sound like content creation, ideation, or innovation theatre. It does not clearly signal accountability for correctness, resilience, safety, tradeoffs, and long-term consequences.

That is why I prefer **Systems Value Engineer**.

The word “Engineer” matters. It says the person is accountable for making value creation repeatable, verifiable, and operationally real.

## What the role actually does

A Systems Value Engineer defines, composes, verifies, and evolves human-agent-software systems so they reliably create measurable business value.

That sounds abstract, but the daily work is concrete.

The role translates ambiguous intent into executable systems. It decomposes goals into constraints, workflows, tests, policies, data contracts, acceptance criteria, and feedback loops. It decides what should be automated, what should be reviewed, and what should remain a human judgement call.

In a software-heavy team, that may still include writing code. But coding is no longer the identity of the role. Coding becomes one mechanism among many.

The role owns the loop from intent to evidence:

1. Clarify the desired outcome.
2. Model the system and its constraints.
3. Generate or modify the implementation.
4. Verify behaviour against tests and policies.
5. Observe the system in use.
6. Capture learnings into reusable institutional knowledge.
7. Improve the system again.

The loop matters more than the artifact.

## The agentic version of the role

AI agents change the mechanics of delivery. They make it easier to create code, documentation, tests, diagrams, migrations, scripts, and analysis. That is powerful, but it also creates a new failure mode: producing plausible output faster than the organisation can verify it.

The Systems Value Engineer is not primarily a prompt writer. Prompting is a low-level interaction technique. The higher-value work is designing the environment where agents can succeed safely.

That means building:

- crisp goals
- small task boundaries
- strong acceptance criteria
- deterministic tools
- machine-readable outputs
- automated tests
- review rubrics
- traceable decisions
- reliable rollback mechanisms
- persistent knowledge capture

In other words, the role is less about asking an agent nicely and more about constructing a system where good work becomes the default and bad work becomes visible quickly.

## The enterprise architecture angle

This title becomes especially useful in enterprise architecture.

Enterprise architecture is often criticised for being too far from execution. It produces principles, target states, capability maps, governance forums, standards, and roadmaps — all useful, but often only indirectly connected to delivery.

A Systems Value Engineer closes that distance.

In an enterprise context, the role asks:

- Which business capability is this system improving?
- Which architectural constraint is non-negotiable?
- Which policy must be executable rather than advisory?
- Which decision needs to be recorded for future agents and humans?
- Which control evidence can be produced automatically?
- Which part of the architecture can be verified continuously?

This is where architecture becomes operational. Not a diagram on the side, but a set of constraints, tests, workflows, and feedback loops embedded in how systems change.

For a bank, this matters. Regulated enterprises cannot treat AI-assisted delivery as a productivity toy. They need traceability, control evidence, segregation of duties, auditability, security, operational resilience, and clear ownership. A better title should reflect that seriousness.

## A practical role definition

Here is the shortest useful definition:

> A Systems Value Engineer engineers sociotechnical systems that turn intent into verified, measurable outcomes.

A slightly more enterprise-ready version:

> A Systems Value Engineer designs, constrains, verifies, and evolves human-agent-software systems so they improve business capabilities while remaining secure, operable, compliant, and economically useful.

That is much closer to the future job than “developer.”

## What changes in hiring and evaluation

If the title changes, the evaluation criteria should change too.

The old evaluation pattern was often:

- Can this person code?
- Do they know the stack?
- Can they solve algorithmic problems?
- Can they ship features?

Those questions still matter, but they are incomplete.

For a Systems Value Engineer, better questions are:

- Can they define a valuable outcome clearly?
- Can they turn ambiguity into constraints and tests?
- Can they design feedback loops?
- Can they use agents without surrendering judgement?
- Can they make risk visible early?
- Can they connect architecture to delivery?
- Can they produce evidence, not just output?
- Can they improve the system of work, not only the work item?

The strongest candidates will still be technically deep. But their value will come from orchestration, verification, judgement, and system design — not raw typing speed.

## Counterpoint: titles are cheap

There is a risk of overdoing this. New titles can become cosmetic. “Systems Value Engineer” only matters if the operating model changes with it.

If the person is still measured only on tickets closed, story points delivered, or lines of code merged, then the title is just decoration.

For the title to mean something, the organisation must also change the work:

- measure outcomes, not activity
- require verification evidence
- make architectural constraints executable
- treat knowledge capture as part of delivery
- give teams tools that support human-agent collaboration
- reward people who improve the system, not just those who push changes through it

The title is useful because it gives language to that shift. It is not a substitute for making the shift.

## What to do next

For a team experimenting with this framing, I would not start by renaming everyone. I would start by rewriting one role description.

Use this as the first draft:

**Title:** Systems Value Engineer

**Mission:** Create measurable business value by designing, constraining, verifying, and evolving human-agent-software systems.

**Accountabilities:**

- translate goals into executable constraints and acceptance criteria
- use AI agents to accelerate implementation while maintaining accountability
- ensure changes are verified through tests, reviews, telemetry, and policy checks
- connect technical work to business capabilities and measurable outcomes
- improve the delivery system itself through reusable tools, patterns, and knowledge capture

**Not accountable for:**

- producing code as the primary measure of contribution
- accepting agent output without verification
- optimising local delivery speed at the expense of system-level value

That is the useful shift: not from developer to prompt engineer, but from software implementation to systems value engineering.
